Ring² Node-Toggle — Implementation Plan

Wiring runtime selection of which producer nodes fire inside an already-captured CUDA graph (cudaGraphNodeSetEnabled) into the Ring² pipeline. This document is a design / plan; it does not modify any existing source.

Branch feature/dmi_kernel_node_toggle · based on the completed investigation in docs/node_toggle_investigation_report.md + the validated probe (docs/node_toggle_probe/probe_dualring_toggle.cu)

✅ Verified measured by the probe on RTX 4090, or already present in current code ⚙️ New code needed the parts to be written ⛔ blocker not achievable with the current structure / sticking point

1 · How current Ring² works (intuitive diagram)

In a single forward step, two lanes advance in parallel and must stay aligned: the GPU data lane (blue) moves tensor bytes, and the host metadata lane (violet) moves shape/dtype/req information. The two lanes are finally rejoined by the p2p thread by FIFO position (the i-th payload ↔ the i-th meta) — this is the linchpin of the entire mechanism.

2 · What changes when toggle is added (change map)

The same diagram, annotated with the fate of each component. In one sentence: the device side is proven by the probe to be naturally compatible (disabling a node leaves no hole; the ring closes up automatically); what actually needs writing is the host-side lockstep and obtaining node handles.

Granularity: what can / cannot be done

Graph nodes are shared by the batch: a node is either on or off for the entire batch. per-request needs a different design (a per-request mask inside the kernel) and is not part of this plan.

3 · Concrete implementation plan (phased, ordered)

Principle: first establish the lockstep discipline where DMI can control its own graph, then tackle the real blocker of framework (vLLM/HF) handles. The device side is already de-risked, so the engineering focus is host lockstep + handle acquisition. Each phase lists: files touched, validation gate, effort (S/M/L), and risk.

Phase 1Lockstep proof through the real consumer (DMI-owned capture) Effort MRisk medium✓ done (C++ E2E)

Goal: where DMI can control the graph, prove end-to-end the §1 discipline "host meta set == device enabled set", with data + correctness flowing through drain/p2p to submit_fn.

Done — as a C++ end-to-end test (tests/ring/test_node_toggle_e2e.cu, 43c626a), not yet wired into HF:

  • 1a · node handles (done). DMI-owned explicit cudaGraph_t capture (begin/end capture + instantiate around the producer launches ourselves) to obtain the cudaGraphNode_t. A sandbox — independent of how HF/vLLM capture.
  • 1b · enabled-set + real toggle (done). cudaGraphNodeSetEnabled flips the set between replays on a non-blocking stream, at step boundaries with the prior replay synced first.
  • 1c · lockstep meta gate (done in the harness). Metas are pushed only for the enabled set; a negative control (push for all nodes) confirms the desync is real (22/37 misaligned). Drives the real RingEngine → drain → p2p → TensorMetaFifo → SubmitFn.
  • 1d · alignment verified (done). Every submitted slice satisfies layer_no == first-byte(payload) and the delivered sequence equals the per-step enabled sets; 0 desync under lockstep.
Gate (met): multiple flips in one run, 0 desync end-to-end through the consumer; negative control trips. Proves the lockstep discipline holds.

Phase 2Toggle-List API — controller + (binding) + capture-time node↦hook registry Effort MRisk low-medium✓ C++ controller done · 2b binding pending

Hard prerequisite. The toggle-list only does anything where DMI owns or is handed both the cudaGraphExec_t and the producer cudaGraphNode_ts. That holds in the DMI-owned sandbox capture (Phase 1) but not in HF/vLLM until handle acquisition is solved (Phase 3). So the controller is handle-agnostic by design; the runtime knob is only usable on a DMI-owned graph or after Phase 3.

  • Done — the controller. ring::NodeToggleController (monitoring/csrc/ring/node_toggle.h): one enabled-set drives both apply(exec)→cudaGraphNodeSetEnabled and the meta-push list. It takes node handles via register_node() — it does not acquire them.
  • Capture-time registry (not runtime observation). Because cudaGraphGetNodes() gives no order guarantee and the payload carries no hook identity, the map is built at capture time: register_node({hook_type,layer_no}, node) in producer publish order. p2p matches positionally, so the same enabled set in the wrong order still desyncs — validate() checks null/duplicate nodes + duplicate hook ids (order itself is a caller contract).
  • §1 invariants are part of the API contract (node_toggle_design_notes.md §1): reconfigure only at a step boundary; apply() only after the prior replay has completed (never mutate a running exec); push metas for the same snapshot the same step; mutate-then-launch. Use apply_and_get_enabled(exec,out) — applies and returns the meta-push list from one snapshot, so no mutation can interleave the two lanes (single-threaded ownership).
  • 2b (pending) — Python binding. Mirror the set_null_mode chain (producer.cu set_ring_null_mode → ring_engine_py → bindings.cpp, with gil_scoped_release): expose set_enabled_hooks([...]) / atomic whole-table set, and feed the controller from RingEnginePy. Needs the backend build to compile-test.
Gate: (done) controller drives the Phase 1 e2e with 0 desync under lockstep, validate() passes; (2b) Python API flips any subset at step boundaries with the meta path in lockstep.

Do NOT do: any re-instantiate / re-capture path (100s µs–ms; avoiding it is the whole point).

Phase 3Framework graph-handle acquisition (was "the real blocker") Effort LRisk medium✓ 3a feasibility proven · 3b wiring pending

Research spike (3a): ANSWERED YES, with a concrete mechanism — docs/node_toggle_phase3_feasibility.md. The "DMI holds no cudaGraphExec_t/node handle" wall is not fundamental; both handles are reachable, proven locally (RTX 4090 / torch 2.10 / vLLM 0.19):

  • Node handles (proven). During capture, right after launching a producer, call cudaStreamGetCaptureInfo and record the tail-dependency node = the node just added — works even though DMI doesn't drive the capture. DMI's producers already run inside the framework's warmup capture, so the hook point exists. (probe_phase3_capture_handles.cu: disabled {1,4,6} of 8 on a foreign exec, interleaved compute intact.)
  • Exec handle (proven). torch.cuda.CUDAGraph exposes raw_cuda_graph() / raw_cuda_graph_exec(); cudaGraphNodeSetEnabled on torch's own exec (via cuda-python) actually toggled a node — probe_phase3_torch_handle.py: a=0 when disabled, restored on re-enable. vLLM's full-cudagraph path uses torch.cuda.CUDAGraph.

3b — real vLLM wiring (PENDING, verify on H100 + backend build):

  • keep_graph=True on the framework CUDAGraph (else torch frees the template after instantiate → node handles dangle) — a vLLM-side setting.
  • Capture mode: feasible on the full-cudagraph path (vllm/compilation/cuda_graph.py); piecewise/inductor cudagraph_trees path TBD.
  • Per-graph registry (one graph per batch size; key node handles by the id from capture info) + §1 lifecycle across vLLM's replay loop; confirm handles stay valid through vLLM's instantiate.
  • Fallback if 3b proves impractical: static superset + global null_mode (v0 already sufficient, investigation report §6).
Gate: 3a (met) — yes + mechanism. 3b — on H100, obtain handles for vLLM's real graphs and toggle a real DMI hook node with the meta path in lockstep.

Phase 1/2 deliverables are independently usable on a DMI-owned/sandbox graph; 3b unlocks the production vLLM path.

Phase 4Adaptive observability (built on the toggle list: cheap default → upgrade on demand) Effort MRisk medium (depends on Phase 2/3)
  • Set the default enabled set to the minimum (e.g. a single-layer residual + final_logits); when a signal is detected (e.g. suspected hallucination), flip the relevant hook-types/layers on at the next step boundary, then flip them back afterward.
  • Built purely on top of the Phase 2 toggle-list state + lockstep gate; touching only a few nodes → cost (#changed)×0.2µs, far below a full-set flip.
  • Decouple the policy layer (when to upgrade) from the mechanism layer (toggle list); start with simple threshold rules.
Gate: The adaptive policy does not break lockstep alignment when flipping at step boundaries; the TPOT overhead is quantifiable (anchored to the paper's ~6%).

The HF path (on top of Phase 1/2) can validate the policy first; online vLLM adaptivity depends on Phase 3 being unlocked.

Explicitly not building (out of scope)

  • ⛔ per-request heterogeneous (#3): graph nodes are batch-shared, so node-toggle physically cannot do it. If needed in the future, it is a separate design (a per-request mask inside the kernel + per-request meta), not node switching.
  • ⛔ any path that requires re-instantiate / re-capture: cudaGraphInstantiate is 100s µs–ms; the entire value of this mechanism is using cudaGraphNodeSetEnabled (0.19µs, no rebuild) to avoid it.
  • ⛔ Modifying the producer_kernel body: the probe has proven it is structurally compatible with zero changes.
  • ⛔ Touching the legacy non-ring set_enabled_hooks (hooks.cpp:207): it is not on the ring/CUDA-graph path and is unrelated to this feature.
