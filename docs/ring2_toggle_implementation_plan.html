Ring² Node-Toggle — Implementation Plan

Wiring runtime selection of which producer nodes fire inside an already-captured CUDA graph (cudaGraphNodeSetEnabled) into the Ring² pipeline. This document is a design / plan; it does not modify any existing source.

Branch feature/dmi_kernel_node_toggle · based on the completed investigation in docs/node_toggle_investigation_report.md + the validated probe (docs/node_toggle_probe/probe_dualring_toggle.cu)

✅ Verified measured by the probe on RTX 4090, or already present in current code ⚙️ New code needed the parts to be written ⛔ blocker not achievable with the current structure / sticking point

1 · How current Ring² works (intuitive diagram)

In a single forward step, two lanes advance in parallel and must stay aligned: the GPU data lane (blue) moves tensor bytes, and the host metadata lane (violet) moves shape/dtype/req information. The two lanes are finally rejoined by the p2p thread by FIFO position (the i-th payload ↔ the i-th meta) — this is the linchpin of the entire mechanism.

2 · What changes when toggle is added (change map)

The same diagram, annotated with the fate of each component. In one sentence: the device side is proven by the probe to be naturally compatible (disabling a node leaves no hole; the ring closes up automatically); what actually needs writing is the host-side lockstep and obtaining node handles.

Granularity: what can / cannot be done

GranularityFeasible?
Global (equivalent to null_mode but actually saves launches)✅
per-hook-type✅
per-layer✅
per-step (flip the whole batch)✅
per-request heterogeneous⛔ not possible

Graph nodes are shared by the batch: a node is either on or off for the entire batch. per-request needs a different design (a per-request mask inside the kernel) and is not part of this plan.

3 · Concrete implementation plan (phased, ordered)

Principle: first establish the lockstep discipline on the HF path (where DMI can control its own graph), then tackle the real blocker of vLLM handles. The device side is already de-risked, so the engineering focus is host lockstep + handle acquisition. Each phase lists: files touched, validation gate, effort (S/M/L), and risk.

Phase 0Standalone node-toggle test target Effort SRisk low
  • Add a test_node_toggle target (and the missing test_null_mode) to tests/ring/Makefile — the current Makefile only has test_rings / test_producer / test_ring_engine; tests/ring/test_null_mode.cu already exists but has no target.
  • Distill the core of the probe (docs/node_toggle_probe/probe_dualring_toggle.cu) into a repeatable CI unit test: link the real ring::producer_kernel + AllocatedRing, capture N producer nodes, disable a subset, and assert that the task ring is contiguous/gap-free with correct offsets.
  • While at it, turn the probe's node→producer-id mapping (enabling each node one by one to build the table) into a reusable helper — Phase 2 will use it.
Gate: Pass the probe-level assertions on H100 (50 random reconfigs, zero corruption), reproducing the 4090 conclusion. Files: tests/ring/Makefile, new tests/ring/test_node_toggle.cu.

Do NOT do: any host/Python plumbing; this phase only locks down the device primitive on the target hardware.

Phase 1HF-path prototype — DMI owns the graph + lockstep meta gate (core proof) Effort MRisk medium

Goal: in an environment where DMI can control the graph, prove end-to-end the §2 discipline "host meta set == device enabled set", with data + correctness flowing through drain/p2p to submit_fn.

  • 1a · Obtain node handles. HF currently goes through torch.compile CompileConfig(mode="reduce-overhead")'s cudagraph_trees (monitoring/generate.py:519,604), and DMI likewise does not directly hold the raw graph. The Phase 1 tradeoff: in the HF prototype, switch to a DMI-owned explicit cudaGraph_t capture (begin/end capture + instantiate around the producer launch ourselves) in order to obtain the cudaGraphNode_t. This is a "sandbox" environment; it is not required to merge with HF's compile path.
  • 1b · enabled-set + real toggle. Maintain an enabled set in the prototype and call cudaGraphNodeSetEnabled to flip it between replays.
  • 1c · lockstep meta gate (the key change). Upgrade the global null_offload gate at ring_transport.py:613-614 into a per-spec enabled check: inside the pre_push_all_metas loop, if not enabled(spec): continue, pushing meta only for currently enabled nodes.
  • 1d · End-to-end alignment verification. Reuse the mechanism in tests/test_e2e_correctness_vs_hf.py: all-on → disable a subset (while the host skips the same subset) → assert that (a) disabled hooks produce nothing, (b) the remaining hooks' act_name/layer_no/shape are all aligned, (c) after re-enabling, the full set realigns.
Gate: Multiple flips within a single run, with zero warnings from the p2p-side :185 size-check and per-hook-correct content landed in ClickHouse. This proves the lockstep discipline holds.

Do NOT do: vLLM integration; do not try to modify HF's compile path itself. This phase is only to "prove the discipline under a controllable graph".

Phase 2Toggle-List API surface — host state + bindings + explicit node↦hook map Effort MRisk low-medium
  • Expose the API, mirroring the set_null_mode chain. The device flag chain is producer.cu:21 set_ring_null_mode → ring_engine_py.cu:93 → bindings.cpp:482. Add the matching toggle-list interface: set_node_enabled(node_idx,bool) / set_enabled_mask(...), exposed to Python via ring_engine_py.{cu,h} + bindings.cpp, with gil_scoped_release (same as set_null_mode).
  • The node→hook map must be explicit. Because cudaGraphGetNodes() does not guarantee order, use the Phase 0 helper: after capture, enable single nodes one by one, observe the published producer-id, and build a node_idx ↦ (hook_type, layer_no) mapping table as the index basis for the toggle list.
  • Single source of truth. The enabled set is shared by two places: the device-side cudaGraphNodeSetEnabled and the host-side pre_push_all_metas filter. Provide an atomic "set the whole table" entry point to avoid a half-flipped state.
Gate: The Python API flips any subset at step boundaries, the Phase 1 e2e assertions still pass, and the node↦hook map is correct for all captured nodes.

Do NOT do: any re-instantiate / re-capture path (that is 100s µs–ms; the whole selling point of this mechanism is to avoid it).

Phase 3⛔ vLLM integration — locate DMI's nodes inside vLLM's captured exec graph (the real blocker) Effort LRisk high (includes unknowns)

This is the real reason the whole feature is stuck. vLLM owns the graph's capture/instantiate/replay; DMI only installs the producer before warmup (vllm_integration.py DMXGPUWorker.load_model), then steers solely via the global null_mode (True during warmup, False afterward, see :172,240-245). DMI holds neither a cudaGraphExec_t nor any node handle.

  • Unknown to resolve: can we enumerate/locate DMI's own producer nodes inside vLLM's captured exec graph and obtain handles? This requires tagging DMI nodes during vLLM's capture flow or adding a hook to recover the handles — there is no such plumbing today.
  • Research directions: does vLLM's CUDAGraphRunner/capture context expose the graph object; is cudaGraphChildGraphNodeGetGraph/node traversal usable; or, at DMI's op registration point, side-channel the stream→node association at the moment of capture.
  • Fallback: if handles cannot be obtained, the vLLM path falls back to a static superset + global null_mode (v0 is already sufficient, see investigation report §6).
Gate (research): answer the binary question "can the handle be obtained" before investing in implementation. Explicitly flagged: feasibility unproven.

Before the gate passes, do not write a toggle implementation on the vLLM path; the Phase 1/2 deliverables are independently usable on the HF path.

Phase 4Adaptive observability (built on the toggle list: cheap default → upgrade on demand) Effort MRisk medium (depends on Phase 2/3)
  • Set the default enabled set to the minimum (e.g. a single-layer residual + final_logits); when a signal is detected (e.g. suspected hallucination), flip the relevant hook-types/layers on at the next step boundary, then flip them back afterward.
  • Built purely on top of the Phase 2 toggle-list state + lockstep gate; touching only a few nodes → cost (#changed)×0.2µs, far below a full-set flip.
  • Decouple the policy layer (when to upgrade) from the mechanism layer (toggle list); start with simple threshold rules.
Gate: The adaptive policy does not break lockstep alignment when flipping at step boundaries; the TPOT overhead is quantifiable (anchored to the paper's ~6%).

The HF path (on top of Phase 1/2) can validate the policy first; online vLLM adaptivity depends on Phase 3 being unlocked.

Explicitly not building (out of scope)

  • ⛔ per-request heterogeneous (#3): graph nodes are batch-shared, so node-toggle physically cannot do it. If needed in the future, it is a separate design (a per-request mask inside the kernel + per-request meta), not node switching.
  • ⛔ any path that requires re-instantiate / re-capture: cudaGraphInstantiate is 100s µs–ms; the entire value of this mechanism is using cudaGraphNodeSetEnabled (0.19µs, no rebuild) to avoid it.
  • ⛔ Modifying the producer_kernel body: the probe has proven it is structurally compatible with zero changes.
  • ⛔ Touching the legacy non-ring set_enabled_hooks (hooks.cpp:207): it is not on the ring/CUDA-graph path and is unrelated to this feature.
