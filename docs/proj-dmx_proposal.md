# proj-dmx

# **Research Proposal**

## **项目标题**

**面向大模型推理的白盒可观测性：神经元级运行态捕捉、存储与查询的端到端研究系统**

## **1. 摘要**

大模型推理已成为关键基础设施，但现有监控系统主要停留在服务侧与高层指标（延迟、吞吐、token 数、错误率等），难以解释推理质量回退、异常延迟、模型行为偏航等问题的内部原因。本研究拟构建一个**端到端、可扩展、面向真实推理引擎的“白盒推理可观测性”研究系统**：在尽量低的运行开销下，捕捉层间计算结果、注意力/MLP 激活、KV cache 等模型内部运行态信号，将其以可查询的结构化形式存入专用存储，并支持研究者对推理过程进行检索、对比与取证分析。研究目标不仅是“能抓到”，更是回答：**在工业推理约束下，什么信号值得抓、如何以可控成本抓、以及如何使这些信号真正改善诊断与理解能力**。

---

## **2. 背景与动机**

### **2.1 问题背景**

- LLM 推理系统复杂度高（批处理、KV cache 管理、融合算子、并行化、编译优化等），导致“现象—原因”的可解释链条断裂。
- 研究工具（例如解释性/Hook 工具链）通常面向离线分析，难以满足在线推理的性能与兼容性需求。
- 因此出现一个明显空白：**推理阶段的模型内部运行态可观测性**，既能提供接近研究级粒度，又能在真实推理场景中可用。

### **2.2 研究动机**

我们认为未来的大模型基础设施需要一种新的“白盒可观测层”：

- 既支持“常态化轻量监控”（持续积累分布与基线），也支持“异常触发深挖”（取证与定位）。
- 将模型内部信号从“实验室里的分析对象”变成“系统运行时的可观测资产”。

---

## **3. 相关工作与研究缺口（Gap）**

1. **高层 LLM Observability（服务侧/请求侧）**：擅长 tracing/metrics，但缺乏对模型内部计算的直接证据。
2. **解释性与 Hook 工具**：能访问激活/中间态，但与生产推理引擎、性能约束、数据治理之间存在距离。
3. **推理引擎 Debug 能力**：通常是临时调试/导出中间张量，不适合作为持续观测与可查询系统。
4. **缺口总结**：缺少一个研究系统去系统性回答
- 在推理引擎约束下，“白盒信号采集”如何做到可控开销？
- 哪些内部信号对诊断最有信息量（information value）？
- 如何把内部信号组织成可检索、可对比、可复现实验的研究资产？

---

## **4. 研究问题与核心假设**

### **RQ1：在真实推理约束下，神经元级/张量级观测能否以可接受开销实现？**

- **H1**：通过分层采集策略（摘要统计 + 触发深挖 + 小流量取证），可以在不显著破坏吞吐/延迟的情况下，获得足够诊断价值的内部信号。

### **RQ2：哪些内部信号对“质量回退、异常延迟、行为偏航”最有诊断信息量？**

- **H2**：并非“抓得越全越好”。少量关键部位（例如特定层的残差流摘要、注意力分布特征、KV cache 行为指标）对定位多数问题具有高性价比。

### **RQ3：如何将内部运行态变成“可查询、可对比、可复现”的研究数据资产？**

- **H3**：若建立统一的信号语义与索引（request/token/layer/version 对齐），并提供查询接口，则可以显著缩短回归定位时间并提升复现率。

### **RQ4：推理引擎兼容性与观测粒度之间的边界在哪里？**

- **H4**：在高度优化（融合算子、图执行等）场景下，“精确逐层全量张量”不可持续；但“可解释的摘要/采样观测”仍然可实现并足够有用。

---

## **5. 研究目标（Objectives）**

1. 设计并实现一个**端到端白盒推理可观测性研究原型系统**（非最终商业产品）。
2. 提出一套**可复用的内部信号语义与采集策略**：常态化/触发式/取证式三档。
3. 给出对关键场景的**实证评估**：性能开销、诊断有效性、兼容性边界与可用性。
4. 形成可公开的研究产出：方法论、基准、系统原型（以及可选的开源子模块）。
