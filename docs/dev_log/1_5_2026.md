## 今日完成:

### 今日目标

- 明确“开启run_with_cache,但是全部非采样 step/request”的性能开销

### 今天做了什么
- [x] 选定“Python gating”作为可落地方案，放弃动态启停 hooks
- [x] 在 `hook_points.py` 增加 `engine.is_capture_enabled()` 的短路逻辑（避免进入 C++ 回调）
- [x] 给 native callback 包一层 `gated_callback`，禁用采样时直接 return
- [x] 在 `run_with_cache`/`get_caching_hooks` 增加采样关闭时的跳过逻辑
- [x] 新增 config 频率验证 benchmark：`HF_Prometheus/benchmark/tests/hf_modified_async_config_benchmark.py`

### 关键结论 / 决策
- 决策：**采用 Python gating**。非采样时只做轻量 Python 判断，不进入 C++ 回调。
- 代价：仍会触发每个 HookPoint 的 Python hook（但不再有 Python→C++ 代价）。

### 方案说明（Python gating）
- 入口：`MonitoringEngine.is_capture_enabled()` 作为统一 gate。
- HookPoint 层：`save_hook` 直接短路；native callback 走 `gated_callback`。
- 缓存收集层：`run_with_cache` 在采样关闭时跳过 `collect_step_futures_into`。
- 预期效果：non‑capture step/request 仍有 Python hook 调用，但避免 C++ 回调开销。
