# 2026-01-04 监控配置设计草案

## 目标
- 提供统一的 config 模块来控制监控引擎行为
- 维度一：深度（hook 哪些点，支持 preset + 自定义列表）
- 维度二：频率（每 N 个 token/step 采样一次）
- 维度三：请求级频率（每 N 个 request 采样一次）

## 现状观察（与设计相关的锚点）
- HookPoint 最终也是 `register_forward_hook`，但 hook 位置是你手工插到模型内部的（`gpt2_p/modeling_gpt2.py`）。
- 目前 `get_caching_hooks()` 已经支持 `names_filter`（list/str/callable），可用于深度筛选。
- `MonitoringEngine` 有 `_current_step_id`，`start_step()/end_step()` 明确了 per-step 边界。

## 设计草案

### 1) 新增独立配置类
建议新增模块：`HF_Prometheus/monitoring/config.py`

```python
@dataclass
class HookSelection:
    mode: Literal["full", "attention", "mlp", "custom"]
    include: Optional[list[str]] = None   # 自定义 hook 名
    exclude: Optional[list[str]] = None   # 黑名单

    def compile(self, hook_names: list[str]) -> list[str]:
        """返回最终启用的 hook 名称列表（用于 native set_enabled_hooks 和 names_filter）。"""
```

```python
@dataclass
class CaptureSchedule:
    step_stride: int = 1      # 每 N step 采样一次
    step_offset: int = 0      # 相位
    warmup_steps: int = 0     # 前 N step 跳过
    capture_prefill: bool = True
    capture_decode: bool = True

    request_stride: int = 1   # 每 N 个 request 采样一次
    request_offset: int = 0   # request 相位
    warmup_requests: int = 0  # 前 N 个 request 跳过

    def should_capture_step(self, step_id: int, phase: Literal["prefill","decode"]) -> bool:
        ...

    def should_capture_request(self, request_id: int) -> bool:
        ...
```

```python
@dataclass
class MonitoringConfig:
    hooks: HookSelection
    schedule: CaptureSchedule
    # 保留 engine 参数（可选）：async_enabled / queue_size / delay_steps / cache_dtype
```

### 2) 深度（hook 选择）如何落地
- 复用 `HookPoint` 的 `names_filter` 机制：`get_caching_hooks()` 接受 list 或 callable。
- preset 的实现只需要在 `HookSelection.compile()` 里根据 hook 名字做规则过滤：
  - `full`：全部 hook
  - `attention`：`hook_q/hook_k/hook_v/hook_z/hook_attn_scores/hook_pattern/hook_result` 等
  - `mlp`：`hook_mlp_in/hook_mlp_out` 等
  - `custom`：直接用 `include/exclude`
- 这样不需要改模型结构，只是控制“启用哪些 HookPoint”。

### 3) 频率（step 级 + request 级）如何落地

**只考虑 C++ 引擎路径（Python engine 忽略）**

- 需要同时支持：
  - step 级采样：每 N step 采样一次（prefill/decode 可区分）
  - request 级采样：每 N 个 request 采样一次

**C++ 侧建议接口**
- `set_capture_schedule(...)`：一次性下发 step/request 的 stride/offset/warmup 参数
- `begin_request(request_id)`：标记 request 边界
- `begin_step(step_id, phase)`：标记 step 边界
- `capture_enabled()`：返回当前是否应采样（由 C++ 结合 request/step 规则判断）

**核心逻辑（C++）**
- request gate：若当前 request 不满足 `request_stride`，则整段 request 内都不采样
- step gate：仅当 request gate 通过时才继续按 step gate 判定
- 具体拦截点（native callback 路径）：
  - `append_hook` / `append_hook_current_step` / `submit_step` 的入口处
  - 若 `capture_enabled == false` → 直接跳过 enqueue/submit

这样可以保证 native callback 永久注册的情况下，也能在 C++ 侧快速短路。

## 集成建议（最小改动顺序）
1) 增加 `monitoring/config.py`（仅数据类）
2) 在 native backend 增加 schedule 状态（request/step 双 gating）
3) 在 `engine_core.cpp` 的 hook enqueue/submit 入口做 fast-path skip
4) 在 Python 侧仅负责传入 config（不做 gating）

## 需要确认的点
- “每 N token” 仍依赖 decode loop 的 step 计数方式；prefill 是否计入 step 需要统一约定。
- request 维度的 ID 来源（服务端入口？benchmark 驱动？）需要统一注入方式。
- hook 名称的 preset 规则是否需要针对不同模型（非 GPT‑2）扩展？

## 运行时接口（示例）
目标：用户在 benchmark 或服务端代码里用最少的 API 接入。

### 1) 初始化配置（一次）
```python
from monitoring.config import MonitoringConfig, HookSelection, CaptureSchedule
from monitoring import MonitoringEngine

cfg = MonitoringConfig(
    hooks=HookSelection(mode="attention"),
    schedule=CaptureSchedule(
        step_stride=4,
        request_stride=10,
        capture_prefill=False,
    ),
)

engine = MonitoringEngine(config=cfg, async_enabled=True)
```

### 2) request 边界（每个请求调用）
```python
engine.begin_request(request_id)
```

### 3) step 边界（prefill/decode loop）
```python
engine.start_step(phase="prefill")
... forward ...
engine.end_step()

engine.start_step(phase="decode")
... forward ...
engine.end_step()
```

### 4) hook 选择（模型初始化时绑定）
```python
hooked_model.monitoring_engine = engine
# run_with_cache(names_filter=cfg.hooks.compile(hooked_model.hook_dict.keys()))
```

### 说明
- `MonitoringEngine(config=...)` 负责把 schedule 下发给 C++ backend。
- C++ backend 在 `begin_request`/`begin_step` 后更新 gate，并在 enqueue/submit 时短路。
