## 一、已完成的核心能力

### ✅ 1. 低开销信号捕捉基础设施（对应 RQ1 - 可接受开销实现）

**已实现（进度 ~70%）**:

#### 1.1 C++ 异步采集管道
- **位置**: `monitoring/csrc/engine_core.cpp` 的 `dispatch_step/process_step`
- **能力**:
  - 非阻塞 GPU→CPU 传输
  - 背压管理（队列满时同步处理）
  - 批量同步策略（单步一次 sync，非 per-task）
  - Pinned memory pool（锁分离、O(1) 释放）
- **性能数据**:
  - Native callback: 0.8s
  - Python fallback: 1.0s
  - Submit 开销: 1040ms → 73ms（14x 加速，commit 4e079e284）
- **优化历史**: SoA → add_task → builder → native callback → pinned pool(被弃用,但是解决pinned-paged内存复用问题)

#### 1.2 分层采集能力
- **常态化**: Native builder 路径（`append_hook`）持续采集
- **触发式**: Global callback 模式
  - API: `set_enabled_hooks(names_iterable)`
  - 运行时启用/禁用特定 hook
  - 延迟写入 cache: `collect_step_futures_into(step_id, cache)`
- **小流量取证**: 运行时控制（虽然缺少显式采样率配置）

#### 1.3 Hook 注入框架
- **位置**: `transformers/src/transformers/models/gpt2_p/modeling_gpt2.py`
- **风格**: TransformerLens 兼容
- **覆盖范围**:
  - Embedding: `hook_embed`, `hook_pos_embed`
  - Layer: `hook_ln1`, `hook_ln2`, `hook_resid_pre`, `hook_resid_mid`, `hook_resid_post`
  - Attention: `hook_q`, `hook_k`, `hook_v`, `hook_z`, `hook_attn_scores`, `hook_pattern`, `hook_result`
  - MLP: `hook_mlp_in`, `hook_mlp_out`
- **总计**: 159 个 hook 点（GPT-2 small）

**缺失部分（30%）**:

#### 1.4 GPU 端摘要统计 ❌
- **提案要求**: "摘要统计 + 触发深挖 + 小流量取证"
- **当前状态**: 只做全量张量传输，无 GPU 端聚合
- **缺失功能**:
  - 激活统计: mean, std, L2 norm, sparsity
  - 注意力统计: entropy, top-K indices
  - KV cache 统计: shape, memory estimate
- **影响**: 这是性能优化的关键缺口，无法实现"高频轻量监控"

#### 1.5 显式采样策略配置 ❌
- **缺失配置项**:
  - `decode_token_stride`: 推理时每 N 个 token 采样
  - `train_step_stride`: 训练时每 N 步采样
  - `layers`: 监控哪些层（当前是全部或无）
  - `heads`: 监控哪些注意力头
- **当前状态**: "要么全抓，要么不抓"

#### 1.6 KV cache 行为指标 ❌
- **提案提到**: "KV cache 行为指标对定位问题具有高性价比"
- **缺失**:
  - KV cache shape tracking
  - Memory footprint estimation
  - Cache hit rate
  - Eviction statistics

### ⚠️ 3. 存储与查询层（对应 RQ3 - 可查询的研究数据资产）

**已实现（进度 ~15%）**:

#### 3.1 基础 cache 结构
- **API**: `run_with_cache(input_ids, engine=engine)`
- **返回**: `(output, cache)` 其中 `cache` 是字典
- **格式**: `{hook_name: BackendFuture/tensor}`
- **位置**: `transformers/src/transformers/models/gpt2_p/hook_points.py:run_with_cache`

#### 3.2 统一信号语义
- **Hook 命名规范**: `blocks.{layer}.{component}.hook_{name}`
- **示例**:
  - `blocks.0.attn.hook_q`: 第 0 层 attention 的 query
  - `blocks.5.hook_resid_post`: 第 5 层残差后输出
- **优点**: 与 TransformerLens 一致，便于迁移

**缺失部分（85%）**:

#### 3.3 专用存储后端 ❌
- **提案要求**: "以可查询的结构化形式存入专用存储"
- **缺失**:
  - 具体数据库取决于Sixian,需要与其数据库进行对接
- **当前状态**: 只是 Python 字典，存储在内存中，无持久化

#### 3.4 索引系统 ❌
- **提案要求**: "request/token/layer/version 对齐"
- **缺失元数据**:
  - 具体需要与Sixian对接
- **影响**: 无法跨请求对比、无法回归分析、无法复现实验

#### 3.5 查询接口 ❌
- **缺失能力**:
  - SQL-like 查询（`SELECT * FROM activations WHERE layer=5 AND metric='mean'`）
  - DataFrame 操作（pandas/polars）
  - 对比分析原语:
    - 不同 request 的同一层激活对比
    - 同一 request 不同层的激活分布
    - 不同模型版本的激活差异
  - 聚合查询（按层/按 head 聚合）

#### 3.6 Schema 设计缺失 ❌
- **建议 Schema**:
  ```
  activations_table:
    - request_id (string)
    - timestamp (int64, unix nanoseconds)
    - model_version (string)
    - step_id (int)
    - layer (int)
    - hook_name (string)
    - metric_name (string, e.g., "mean", "std", "raw")
    - value (varies: float for stats, blob for raw tensor)
    - metadata (json, e.g., {"batch_size": 8, "seq_len": 128})
  ```
