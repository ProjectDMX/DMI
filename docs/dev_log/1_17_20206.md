# 1_17_20206 监控引擎线程/进程与 CacheFuture 行为分析

## 背景
新接口改为 `host_engine.enqueue(cache_dict)`，其中 `cache_dict` 是 `str -> CacheFuture` 的字典。  
关注点：CacheFuture 是否能跨进程传递？Python backend / C++ backend 的线程关系如何？

## 结论（简述）
- **CacheFuture 不能安全跨进程传递**：内部包含 `threading.Event`、Python 对象引用以及 native backend 句柄/token。  
- **推荐同进程 + 多线程**：host_engine 在同一进程内用线程消费 `cache_dict`。  
- **Python backend：** 由 `_PythonBackend` 的 worker 线程 `set_result()`  
- **C++ backend：** 不靠 `threading.Event` 通知完成，而是通过 `backend.future_*` 在 C++ 侧等待/返回结果

## 关键代码证据
### 1) CacheFuture 内部结构（包含 threading.Event）
文件：`HF_Prometheus/monitoring/task.py`
```python
class CacheFuture:
    __slots__ = ("_task", "_event", "_result", "_exception", "_backend", "_token")

    def __init__(self, task: MonitoringTask):
        self._event = threading.Event()
        self._backend = None
        self._token = None
```
说明：`threading.Event` 与 Python 对象引用不可跨进程直接序列化；`_backend` 是 native 后端对象句柄，也不可在进程间共享。

### 2) Python backend 的线程关系
文件：`HF_Prometheus/monitoring/engine.py`
```python
def _worker_loop(self) -> None:
    ...
    for task, future in work.tasks:
        tensor = _process_task_sync(task, cache_dtype=self.cache_dtype)
        future.set_result(tensor)
```
说明：Python backend 下由 **Python worker 线程**执行任务并 `set_result()`，`CacheFuture._event` 用于唤醒 `result()/wait()`。

### 3) C++ backend 下 CacheFuture 的行为
文件：`HF_Prometheus/monitoring/task.py`
```python
def result(self, timeout=None):
    if self._backend is not None and self._token is not None:
        result = self._backend.future_result(self._token, timeout)
        self._event.set()
        self._backend = None
        self._token = None
        return result
```
说明：C++ backend 不靠 Python 线程 `set_result()`；`result()` 直接调用 native `future_result()` 阻塞等待，拿到结果后才触发 `_event`。

### 4) C++ backend 绑定 token 的位置
文件：`HF_Prometheus/monitoring/engine.py`
```python
tokens = backend.submit_step(step_id, task_specs, stream_handle)
for token, (_, future) in zip(tokens, tasks):
    future.bind_backend(backend, token)
```
说明：native backend 返回 token，绑定到 CacheFuture；后续 `future.result()` 将通过 native backend 查询该 token。

## 线程关系总结
- **Python backend**：  
  - 一个 Python worker 线程处理任务  
  - `CacheFuture` 通过 `threading.Event` 完成通知  
- **C++ backend**：  
  - native backend 自身有 C++ worker 线程  
  - `CacheFuture` 仅保存 token；`result()` 调 `future_result()` 获取结果  
  - `threading.Event` 只是本地缓存用途，不是 native 完成通知

## 对新 API 的影响
`host_engine.enqueue(cache_dict)` 如果直接收到 `CacheFuture`：
- 必须在**同一进程**内消费（线程安全可控）  
- host_engine 线程调用 `future.result()` 即可等待并拿到 tensor  
- 不建议跨进程传递 CacheFuture（不可 picklable）

## C++ 后端流程细节：token 是什么？如何等待完成？
### token 含义
- token 是 **native backend 为每个 task 分配的唯一 ID**  
- 用于在 C++ 侧 `slots_` 里索引对应的 `ResultSlot`

**token 分配位置（C++）**  
文件：`HF_Prometheus/monitoring/csrc/api_submit.cpp`
```cpp
int64_t token = next_token_++;
tokens.push_back(token);
auto slot = std::make_shared<ResultSlot>();
slots_.emplace(token, std::move(slot));
entry.token = token;
```

### 结果就绪与等待机制
1) C++ worker 完成 task 后写入 `ResultSlot` 并通知  
文件：`HF_Prometheus/monitoring/csrc/engine_core.cpp`
```cpp
void store_result(int64_t token, at::Tensor&& tensor) {
  auto slot = get_slot(token);
  {
    std::lock_guard<std::mutex> lock(slot->mutex);
    slot->tensor = std::move(tensor);
    slot->ready = true;
  }
  slot->cv.notify_all();
}
```

2) Python 侧 `CacheFuture.result()` 会调用 native `future_result()`  
文件：`HF_Prometheus/monitoring/task.py`
```python
result = self._backend.future_result(self._token, timeout)
```

3) native `future_result()` 会等待 `slot->ready`  
文件：`HF_Prometheus/monitoring/csrc/api_submit.cpp`
```cpp
slot->cv.wait(lock, [&] { return slot->ready; });
```

**结论**：C++ backend 下没有 Python 线程去 `set_result()`；  
`CacheFuture` 通过 `future_result()` 与 C++ 的 `ResultSlot` 直接同步。

## cache_dict 里是否包含 logits？
当前实现：**默认不包含**。  
- `cache_dict` 只保存 hook activations（`CacheFuture` / `BackendFuture`）  
- logits 在 `outputs` 里（例如 `GPT2LMHeadModel` 返回 `outputs.logits`）

如果需要把 logits 也进 `cache_dict`，必须显式加入，例如：
```python
cache_dict["final_logits"] = outputs.logits
```
或在 backend 侧加一个额外 task，把 logits 包装成 `CacheFuture`。
