DMI Runtime Node-Toggle

Design · Usage flow · Expected performance — a showcase of the Ring² post-capture producer-node toggle

What it is, in one line: flip which producer / hook nodes fire inside an already-captured CUDA graph, between replays, via cudaGraphNodeSetEnabled — no re-capture, no re-instantiation.

Measured RTX 4090, isolated microbench (real producer + dual-ring) Projected online vLLM+DMI TPOT, anchored on the paper's measured ~6% full-capture overhead Not supported outside what this mechanism can do

1 · What it is — the design

The single source of truth drives two lanes in lockstep

A graph captures the superset of hook nodes once. At runtime, one enabled-set is the single source of truth that must drive both lanes identically — the device lane (which kernels run) and the host lane (which metadata is pushed). If the two ever disagree, the pipeline desyncs.

◆ Single source of truth — the enabled-set (this step: resid_pre·L0, resid_final, logits)
resid_pre·L0 attn_scores·L0 q·L3 k·L3 resid_final final_logits

Each chip = one hook node (a tensor @ a layer). Green = on this step, red = off this step (runtime state, freely flipped back — not "unsupported").

↓   the SAME set feeds both lanes   ↓
⬢ Device lane — which kernels run
apply_toggle() → cudaGraphNodeSetEnabled
For each captured producer node, set its enabled bit on the instantiated cudaGraphExec_t. Disabled nodes are skipped at launch dispatch — the kernel never runs, so it never advances the ring head. The ring "closes up": remaining producers publish a contiguous, gap-free run of task entries.
ring_engine_py.h · node_toggle.h::apply()
⬡ Host lane — which metadata is pushed
pre_push_all_metas → is_hook_enabled() gate
Push a TensorMeta only for hooks that are enabled this step. The per-spec is_hook_enabled() gate replaces the old global null_offload skip. The number of metas pushed shrinks to exactly match the enabled producers.
ring_transport.py · ring_engine_py.h::is_hook_enabled()

Why the two lanes must agree: p2p matches positionally

The p2p thread rejoins the lanes by FIFO position: the i-th drained payload is paired with the i-th popped meta (fifo_.pop(meta)). So the enabled node-set and the pushed meta-set must be the same set, in the same order, the same step.

Capture-time registry — how nodes get their handles

During capture, the producer op records its own kernel node via cudaStreamGetCaptureInfo (reading the tail dependency = the node just added). Handles are kept in a registry keyed per cudaGraph_t, registered in producer publish order (cudaGraphGetNodes() gives no order guarantee, so enumeration order is never used).

Requires keep_graph=True on the framework's CUDAGraph so the template survives instantiate() — otherwise the recorded node handles dangle (proven: toggle returns err 1 without it).

The "enabled AND captured" guard

A hook is treated as on only if it is enabled and was actually captured as a node. This guarantees a meta is never pushed for a hook that has no captured payload behind it — keeping the host lane from ever over-counting relative to the device lane.

enabled×captured=push meta + run node
enabled×not captured=no meta (safe)

2 · Usage flow — the lifecycle

Three points in the lifecycle: turn on capture recording before the graph is captured, bind each exec after instantiate, then reconfigure at every step boundary.

Capture time — record the nodes
engine.enable_toggle_capture(true) before the graph is captured. Each producer op then records its kernel node into the per-graph registry as it fires during capture.
After instantiate — bind the exec (per captured graph)
engine.bind_graph_exec(raw_cuda_graph(), raw_cuda_graph_exec()) associates the instantiated exec with the graph the nodes were recorded under. One bind per captured graph (vLLM keeps one per batch size).
Each step boundary — reconfigure from one source
transport.set_active_hooks(enabled) drives the device toggle (apply_toggle → cudaGraphNodeSetEnabled) and the host meta gate (is_hook_enabled) from the same enabled-set. Call only when the prior replay has completed.

The §1 invariants — hard rules for every reconfigure

1
Reconfigure only at a step boundary. Never mid-step / mid-forward.
2
Prior replay must be complete before mutating the exec graph. Calling cudaGraphNodeSetEnabled while the exec may still be running is undefined behavior.
3
Node-enabled set ≡ host meta-push set — same version, same step, derived from one source of truth, never independently.
4
Mutate the exec graph, THEN launch. Order: prior replay done → apply SetEnabled for the new set → push matching metas → launch. Not interleaved.
5
Never modify the enabled bit while the exec is running (corollary of #2, stated explicitly — it is the UB that bites silently).

Violating #3 desyncs the positional meta↔payload matching in p2p_thread.cpp — corrupting every subsequent hook's association.

The 7-method API surface (RingEnginePy, ring_engine_py.h)

3 · Expected performance

Read this first. The microsecond figures below are from an isolated microbench with no model compute (synthetic 1MB tensors, self-captured graph). The point is the relative takeaways — true-disable is ≈ free of the producer cost, null_mode still pays the launch, and reconfigure is ≈ free — not a serving overhead number. Serving-% is the separate projection further down.

The true-disable vs null_mode delta is exactly the launch overhead null_mode cannot remove (it still launches every producer, then early-returns): ≈ 0.6 µs/node → 9.8 µs @16 nodes, 91 µs @145 nodes. This is the concrete, scale-dependent value of true-disable over the existing soft switch.

Reconfigure is cheap, linear, and crucially avoids re-instantiation (a cudaGraphInstantiate rebuild is 100s of µs to ms — the cost this mechanism deliberately sidesteps). Touching a few nodes for adaptive escalation is essentially free.

Per-request heterogeneity Not supported

Different requests in the same batch capturing different tensors — node-toggle cannot do this. Graph nodes are batch-shared: a node is on or off for the entire batch. It would need a separate mechanism (a per-request mask inside the kernel + per-request meta) and is out of scope for the toggle list.

Status: device mechanism (cudaGraphNodeSetEnabled ring-toggle) + host meta gate + the NodeToggleController + lockstep meta gate are built & verified locally end-to-end (RTX 4090, through the real RingEngine → drain → p2p → TensorMetaFifo → SubmitFn consumer, with a negative control proving necessity).
