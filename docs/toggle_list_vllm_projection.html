DMI Toggle List — vLLM 下的性能与用例预计

运行时切换「哪些 producer/hook 节点在已捕获的 CUDA graph 里发火」—— 通过 cudaGraphNodeSetEnabled 在两次 replay 之间翻转,无需重新 capture。

实测 RTX 4090 / CUDA 13 微基准(真实 producer + dual-ring) 预计 推算到 vLLM+DMI / H100 在线服务(锚定论文 ~6% 在线 TPOT) 不支持 此机制做不到

1 · 什么是 Toggle List

一张「节点开关表」:graph 捕获时把全集的 hook 节点都录进去(superset),运行时给每个节点一个 on/off 状态,按需在 step 边界翻转。两条硬约束:

resid_pre·L0 attn_scores·L0 resid_final q·L3 k·L3 final_logits → replay

每个方块 = 一个 hook 节点(某张量 @ 某层,均为 DMI 支持的类型)。 绿色 = 本步开启捕获, 红色 = 本步关闭(运行时状态,可随时翻回,不代表不支持)。 此例:这一步只抓 resid_pre·L0 + resid_final + final_logits。

① 设备侧:环会自动「合拢」 实测

被关掉的节点不发火,就不推进 task_head/payload_head(head 由 kernel 运行时自己推进,非 host 预留),剩下的节点写出连续无空洞的 task entry。50 次随机翻转零损坏。

② 主机侧:meta 必须 lockstep 待实现

Python 的 pre_push_all_metas 必须只为开启的节点推 meta,和节点开关同步。否则 host TensorMetaFifo 与 payload 按位置错配 → 全盘 desync。这是软件纪律,非环架构限制。

可行性门槛(blocker): vLLM 拥有 CUDA graph,DMI 目前拿不到节点句柄(只靠全局 null_mode 软开关)。要在真实 vLLM 下做 toggle list,需新增「在 vLLM 捕获的 exec graph 里定位 DMI 节点」的管线 —— 建议先在 HF 路径(DMI 可自持图)落地原型。

每请求异构监控 (#3) 不支持

同一 batch 里不同请求抓不同张量 —— node toggle 做不到:graph 节点是 batch 共享的,一个节点要么整批开、要么整批关。需另设机制(kernel 内 per-request mask + per-request meta),不在 toggle list 范畴。

6 · 预计的假设与边界(诚实说明)

  • 锚点:「全量捕获 ~6% 在线 TPOT」来自 DMI 论文实测;其余 TPOT% 由该锚点 + 微基准 toggle 成本推算,非真实 vLLM 端到端实测。
  • TPOT 基线假设几 ms 量级(H100,4B–14B,中等 batch);具体百分比随 TPOT、hook 数、张量大小浮动。
  • hook 数以 145(backpressure 配置)为「多 hook」代表;少 hook 时全量翻转成本更低(线性)。
  • 微基准为隔离测量:合成 1MB 张量、自持 graph、无模型计算、无 Python meta-FIFO / drain / ClickHouse。设备侧一致性与相对开销可信;端到端服务开销待 H100 实测。
  • 未计入:vLLM 取节点句柄的工程成本(blocker)、meta lockstep 落地、连续 batching 下的调度交互。

下一步可把这些预计变成实测:在 HF 路径落地 toggle list + meta lockstep 原型,然后在 Zaratan H100 上按真实 hook 配置量化 idle / 单探针 / 自适应三档的 TPOT。详见 docs/node_toggle_investigation_report.md §5。
